BOTS PLAY GAMES #1
"BLACKSMITH HIT"

First part of the series where we try and breakdown some games and write individual bots to play them.
 

Foreplay

What is BlackSmith Hit?

Blacksmith Hit is a PC/Mobile game with a pretty simple gameplay mechanic. It has both single player and multiplayer modes and plenty of characters to unlock. The art style is flat and vector-ish and looks amazing. For PC you can buy it here.

  • In this series, I'll try and break-down some classic games and write some algorithms which will be able to play the games. 
  • But there is a catch , I won't be writing any code (at least for now), so YOU will be doing all the badass implementations.
  • Once the series is complete , I'll try and put in the implementations for all the games in Python but don't wait for that, implement it yourself! 
  • Now coming back to the game , the mechanic is pretty simple. You have blocks of iron infront of you , coming one after another. You press "A" if the iron block infront of you is hot (red) , or you press "D" if the block infront you is cold (grey). And there is a time constraint so you have to give the inputs quickly or the iron cools down and you loose.(basically a countdown timer which ticks depending on the frequency of your input and the higher you score , the more frequent your inputs should be , for you to stay alive).

The main part!

BOT PIPELINE

So this is basically how the bot works , and you're the wizard who'll be writing that magic part.

So let's get to business. We'll start with the basics of the bot. How will the bot work?

Input : We somehow have to give the algorithm the data , so that it'll give you an action to perform depending on the input you give. It's a basic look-up table in our case as we're not kicking in any unsupervised machine learning(yet!). So we somehow have to give the algorithm the TYPE of the block infront of the player. You tell your algorithm whether the block is hot or cold. How do you do that? Few months I was trying to work my way around computer vision and this was one of my initial practice programs I wrote while learning. What I did was , I set my game to "windowed mode" , ran my python code in the background because the advantage here is , the point of action in this game is always at one place i.e the current iron block infront of the player , so all I had to do was, get a pixel's value in that region to know if it's red or grey (Hot or cold).

Magic : Now once you have your input , * drumroll* all you have to do is write an if-else condition. If the color is red - do action (1) else do action (2).

Output : Python's "pyautogui" didn't work for me while giving the input to a game , so I had to use some other library which lets you give input in DirectX context. So depending on the color of the block , if it's red , the output is given as 'Keypress A' to the game window , else it is given as 'Keypress D'.

Note : But there is something to look out for, there should be a wait time for every iteration of your little pixel color checking because, in the game there is a super small delay for the current block to dissappear and a new block to come which is obvious , so keep that in mind while you're implementing it.

Now , your job is to get a pixel's R value (out of RGB) from that "point of action" as input and feed it to your if-else. You can do that using any image processing libraries which have straight forward functions to get a pixel's RGB values.

Climax

Now your program should be able to play the game. Implementing this won't get you a job in Google , but atleast feel good that something you wrote is capable of playing a game by itself. As this is the first part , we kick this off with a pretty simple game where you only have 2 decisions to make and one parameter to decide upon. 

By the way, the game is free for android , so if you don't want to buy it just to write a stupid bot , download an android emulator like nox ( not a paid promotion ) , map your input keys and let the bot do the job.
