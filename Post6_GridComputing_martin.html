MARTIN'S ALGORITHM *DISTRIBUTED COMPUTING IN A GRID*

What the heck is Martin's algorithm? And how is it used in a distributed grid network?
 

Foreplay

Martin's algorithm
As we already talked about how the resource sharing and synchronization is acheived in a grid for distributed computing in the last post, Martin's algorithm is one of those synchronization protocols where the perfect sync. is achieved using a token.
  • In Martin's algorithm , atleast the bare bones version of it , uses the topology of a ring where all the nodes are arranged in the form a ring , not physically but logically. 
  • As we know, in the token based protocols , for a node to enter its critical section , it has to have the token. This token is moved in one direction of the ring , traversing through all the nodes one by one. When the token enters a node, then and only then , that  node can execute its magical operations on the shared resources.
  • But there is something called request , where nodes can request to enter the critical section. The requests travel in the opposite direction of the token.

The main part!

For example , consider 5 nodes A, B, C, D, E arranged in a logical ring. Consider C and D put a request to enter their critical sections. So the request of C goes to B, B checks if there are any pending requests with it. If there are no pending requests in B , it forwards the request to A which holds the token. Similarly , D's request goes to C. But C already has a request which is it's own, so the request doesn't move forward. When the token circulation initiates, the token first goes to B , then to C. C enters its critical section and then sends it to D. Then enters its critical section and moves the token forward.

Algorithm :

For each node in the network,

    1) If you need to enter critical section, request your predecessor. Then for the predecessor - if there are no pending requests with it, you forward the request to the next predecessor and this goes on till the request reaches a node either with the token or pending requests. The circulation of the request stops there.

    2) Token is given to the successor, enter into critical section if needed , else just pass it on to the successor. 


Note : One can only enter into critical section if they put a request for it. So from the above example , as B did not request for a critical section , it just passes on the token to C directly as it also has no pending requests.

Climax

So that is how Martin's algorithm works in the token based resource sharing context. Reference notes is taken from a book called "Fundamentals of grid computing". Next post in this pipeline will be on another token based protocol which is Naimi-Trehel.
